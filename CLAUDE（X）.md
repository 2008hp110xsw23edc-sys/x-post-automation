## このファイルの設計思想

**共感+アドバイス型の強みを引き継ぐ**
- 職場HSPの「あるある」を箇条書きで言語化して共感を集める
- 「これはHSP特性だから」という自己理解の気づきを与える
- 明日から動ける具体的解決策を最後に届ける

**本音+問題提起型の強みを引き継ぐ**
- 「これ本気で言います」「断言しますが」の本音フックで最初の1行を作る
- 職場HSPが消耗する「構造的な理由」を本質からえぐる
- 逆説的な気づき（停滞・消耗の意味を変換する）で終わらせない

**基本方針**

- すべての投稿は**1投稿完結**で書く
- 「本音フックで引きつけ → 共感 → 気づき → 解決策」の要素を1投稿の中に凝縮する
- どの形式を使っても「読んだ人が動ける」状態にして終わらせる

## コンテンツのアプローチ・トーン

* **文体**: 柔らかく、包み込むような語り口。「〜です」「〜ます」調を基本にする
* 職場HSPが「薄々感じていた本質」を言語化して「そうだった！」を引き出す
* 問題提起→本質の指摘→「じゃあどうする」の流れを基本にする
* 励ましの押しつけはしない。「気づいた人は動ける」という信頼ベース
* 箇条書きで職場HSPの特徴・シチュエーションを具体的に列挙する
* ポジティブな締めは「前向きすぎない」トーンで。「少しずつでいい」感覚
* 解決策パートは「私はこうしてきた」スタイルで上から目線にしない
* 改行を多用し、スマホで読みやすくする
* 箇条書きは番号付き（①②③）と「・」を使い分ける

### 発信のポジション
「同じ職場で消耗しながら、なぜこんなに辛いのかを考え続けてきた仲間」として発信する。専門家でも成功者でもなく、「本質に気づいた一歩先の同志」のポジション。

### 本音フックで最初の1行を作る
冒頭の1行が勝負。スクロールを止める言葉を選ぶ。

使えるフックワード：
- 「これ本気で言います」
- 「断言しますが」
- 「綺麗ごと抜きで話しますね」
- 「ちょっと聞いてほしいです」
- 「気をつけてください」（マヒサイン系）
- 「〇〇な人、これ当てはまりますか？」（あるある系）

### 職場HSPの「構造的な問題」を言語化する
個人の弱さではなく、環境・仕組み・HSP特性のミスマッチという視点で語る。特に扱うテーマ：
- 威圧的な上司の言葉が一日中頭に残る理由
- 怒りを出せない・本音を言えない理由
- 真面目にやっているのに評価されない構造的な理由
- 我慢→消耗→メンタル崩壊のループが起きるメカニズム
- 心が壊れ始めているサイン（職場版）
- 退職・転職を言い出せないHSP特有の葛藤

### 職場HSPあるある中心
職場に特化した「あるある」コンテンツを中心に据える。特に反応が取れるテーマ：
- 職場の威圧・圧力に対する感じ方の違い
- 本音が言えない・怒りが出せない理由
- 3人以上の会議・グループコミュニケーションの辛さ
- 退職・転職を言い出せないHSP特有の葛藤
- 心が壊れ始めているサイン
- 真面目にやっているのに評価されない悔しさ

### 具体的解決策を必ず最後に届ける
「わかった、でもどうすればいい？」に答えることがこのアカウントの核心。テーマ別に以下を使い分ける：

**人間関係・上司への対処**
- 即答しない。「少し考えます」で場を外すルールを作る
- 会話後すぐメモで感情を吐き出してリセットする
- 席・ランチ・休憩のタイミングを意図的にずらして物理的距離を作る

**感情コントロール・心の管理**
- 怒りや傷つきはその場では飲み込み、帰宅後に紙に書き出して手放す
- 1日の「刺激量」を把握し、消耗したら意図的に無音・ひとり時間を確保する
- 「今日のしんどかったこと」を就寝前にスマホにメモして完結させる

**評価・キャリアの悩み**
- 成果を小さく記録する（自分評価シートを作る）
- HSPに向いている職場の特徴を転職の軸にする
  （少人数・専門職・裁量がある・競争より協力の文化）

**退職・転職の一歩**
- 退職を言い出せないなら退職代行サービスを選択肢に入れる
- 転職エージェントは複数登録し「情報収集目的」で面談する（決断は後でいい）
- まず有給消化・休職で「距離を取る」ことを最初のステップにする

**日常の心の守り方**
- 職場では「8割の自分」で関わる（全力で共感しない練習）
- 週1回、自分だけのリセット時間をスケジュールに入れる
- 「ここが辛い」をXか信頼できる人に吐き出す場を作る

### 全体フロー

```
【毎日 2:00 JST】
① analyze_reference_accounts.py
   参考アカウント.txt の3アカウントの直近30日間の高エンゲージメント投稿を分析
   → バズる冒頭フック・テーマ・文体パターンを抽出
   → 分析/reference-insights-latest.json に保存

② generate_posts.py
   ①の分析インサイトを反映し、6タイプ×1投稿=計6投稿を生成
   → posts/YYYY-MM-DD.md に保存

③ GitHub にコミット・プッシュ
   → 確認・修正 → X に投稿

【毎週土曜 5:00 JST】
④ analyze_posts.py
   自分のアカウントの直近7日間メトリクスを取得・分析
   → 分析/YYYY-MM-DD.md + posts/YYYY-MM-DD-analyzed.md に保存
   → 以降の投稿生成に反映
```

### 投稿生成の基本方針

参考アカウント（`参考アカウント.txt`）の直近30日間の高エンゲージメント投稿を分析し、その結果を起点に6投稿を生成します。

**生成の優先順位**

1. **参考アカウント分析を最優先で反映する**
   - バズった冒頭フックのパターンを取り入れる
   - エンゲージメントが高かったテーマ・切り口を選ぶ
   - 文字数・絵文字・改行など文体のコツを踏襲する
   - 「★ 即日反映すべき改善ポイント」は必ず取り入れる

2. **分析から導き出した6投稿を生成する**
   - 分析で見えた「反応が取れるパターン」をベースに投稿タイプを割り当てる
   - 問題提起型・あるある型・本音告白型・マヒサイン型・逆説気づき型・解決策型の6タイプを各1投稿、計6投稿を生成する
   - タイプの順番より「分析から何が刺さるか」を優先してテーマを選ぶ

### 投稿フォーマットのルール

**構成は参考アカウント分析から導く**

固定フォーマットは使わない。参考アカウントの高エンゲージメント投稿から抽出した「バズる構成パターン」を最優先で使う。

- 分析で判明した冒頭フックの型をそのまま採用する
- 分析で反応が多かった構成（列挙・問いかけ・告白など）を投稿タイプに合わせて選ぶ
- 分析結果がない場合のみ、下記の基本ルールをデフォルトとして使う

**基本ルール（分析結果がない場合のデフォルト）**

- 冒頭は「スクロールが止まる」フックで始める
- 1投稿完結・140字以内
- 「〜です」「〜ます」調を基本に、「〜ですよね」「〜なんです」で柔らかく話しかける
- 短文を積み重ねるリズム感（1文1〜2行で改行を多用）
- 箇条書きは番号付き（①②③）または「・」で統一する
- 絵文字は1投稿1〜2個
- 解決策は1〜2つに絞る
- ハッシュタグは原則なし

**禁止事項**

- 上から目線のアドバイス・説教口調・「〜すべき」の押しつけ
- 成功自慢・マウント的な表現
- 根拠のない「大丈夫！」「絶対できる！」の連発
- 「気にしすぎ」などHSPを否定する表現
- ハッシュタグは原則なし（告知型のみ使用可）
